# Scope cascade precedence for auth and headers

Collections and Folders both carry scope config (auth, default headers, vars) that
Requests inherit with precedence request → folder → collection; folders stay flat
(single level) and their config lives inline in `collection.yaml` as `folders` objects,
with bare strings still accepted for backward compatibility. Two collision rules are
deliberate: headers merge by case-insensitive name with the nearer scope winning (and a
Request may disable an individual inherited header outright), but a header generated by
the **resolved auth method overrides same-named default headers from any scope** — even
a nearer one — because auth is the declared intent of "how I authenticate"; only a
header typed on the Request itself beats it. A pure scope-precedence rule for the
auth-vs-header collision was rejected as producing case matrices no one remembers.
Folders get no `baseUrl`; URL prefixes are expressed with folder vars instead, to avoid
introducing a second URL-composition semantic (override vs concatenate).
